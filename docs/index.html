Baraja

A shared deck app that behaves like a physical table: shuffle, deal, give, take, peek, show, flip, cut piles, animate the motion, and run through either a cloud room or a dealer-hosted local room.

View the architecture Read design details

Understanding

The Product Problem

Baraja is a deck simulator, not a rules-enforcing card game. It should mirror a physical table where people can attempt real deck actions, the app records what happened, and visibility rules keep private cards private.

01

Shared State

Every participant sees the same table, piles, hands, and card movement order. The source of truth is authoritative, not derived independently on each device.

02

Physical Deck Model

The engine validates state, ordering, and card existence, but social rules decide whether a move was acceptable. Give, take, peek, and show are first-class actions.

03

Dynamic Visibility

Face-down cards are not merely hidden in the UI. Each user receives a projected state view based on who currently knows each card value.

04

Local And Online Play

Cloud rooms support durable online sessions. Dealer-hosted rooms allow in-person play without internet by making the dealer device the local authority.

Options Considered

Three Ways To Host The Table

Cloud and dealer-hosted rooms are both part of the v1 direction. Fully peer-to-peer hosting is rejected for now because ordering, discovery, and recovery become much harder.

Selected as remote mode

Cloud Hosted

A backend service owns the deck state, receives commands, validates state transitions, and broadcasts projected updates to every client.

  • Best for remote play and reconnects.
  • Strongest practical privacy boundary.
  • Requires internet and hosted infrastructure.
Selected as offline mode

Dealer Hosted

The dealer device runs the room authority locally. Other devices join over local Wi-Fi, hotspot, or a nearby transport.

  • Works without internet for in-person games.
  • Maps naturally to the dealer role.
  • Dealer device becomes the session dependency.

Fully Peer To Peer

Every participant shares responsibility for state, ordering, discovery, and conflict resolution with no single authority.

  • Most decentralized.
  • Highest complexity for ordering and trust.
  • Hardest to make consistent across web and native.

Selection

Use One Game Engine With Two Hosting Modes

The selected design keeps the card rules, visibility logic, and animation events shared. Cloud and dealer-hosted play are authority adapters over the same command, event, and projection model.

The authority implementation, transport, and persistence strategy change by mode.

The game engine, commands, events, animations, versioning, and projection rules stay shared.

Design Details

The Shared Engine

The engine models cards, zones, dynamic visibility, commands, and events. This keeps gameplay behavior independent from whether the room is hosted in the cloud or on the dealer's phone.

Product Stance

Physical Deck With Social Accountability

Baraja does not try to enforce every game rule. Any player may flip, move, or cut table zones. Give and take are explicit cross-hand actions: in v1, a player can take from another player's hand without an app-level consent gate. Hand ownership still controls default visibility and private arrangement. Public events and table etiquette are the enforcement layer.

Zones

A deck, a player hand, a discard stack, and a table pile are all ordered zones. Each zone stores an ordered list of card ids, so shuffles and cuts replace a list rather than rewriting a numeric order on every card.

Deal

Moves cards from a source zone to a destination with face-up or face-down intent.

Shuffle

Randomly reorders a zone and resets each card's viewers from that zone's defaults.

Give

Moves a card from the actor's zone to another player, table zone, or pile.

Take

Moves a card from another zone to the actor's zone. This is allowed in v1 and publicly recorded.

Peek

Grants visibility inside the current zone and broadcasts who peeked at whom.

Show

Widens visibility inside the current zone to one player, several players, or everyone.

Flip

Toggles one or more table cards face up or face down and recomputes viewers from face state.

Cut

Reorders a pile by splitting it at a random or chosen point and recombining the halves.

Core State Model

Visibility is a per-viewer set, not a static card category. Projection includes a card value only when the requesting player is currently in the viewer set or the value is visible to everyone. Whenever a card moves zones, its viewers reset from the destination zone's defaults. 

type ViewerSet = "everyone" | Set<PlayerId>;

type CardState = {
  id: string;
  value: CardValue;
  zoneId: ZoneId;
  face: "up" | "down";
  visibleTo: ViewerSet;
};

type ZoneState = {
  id: ZoneId;
  type: "deck" | "hand" | "table" | "pile";
  ownerPlayerId?: PlayerId;
  cardIds: CardId[];
};

Peek Notification Contract

Peek is a privacy primitive, not a private UI affordance. The social cost only works if the whole table can see that it happened.

  • Every peek emits a public event naming the peeker and the target.
  • Peek events survive reconnect as part of the game event log.
  • Analytics must not make peek history queryable outside the room context.

Command Ordering

  1. Client sends a command with room, actor, and expected version.
  2. Authority rejects stale versions and invalid card references.
  3. Authority mutates state atomically.
  4. Client reconciles from the latest snapshot after rejection.

Animation Path

  1. Authority emits semantic events, not screen coordinates.
  2. Each client computes local card positions for its viewport.
  3. Cards fly from source zone to destination zone with staggered timing.
  4. The final layout settles after the animation completes.

Offline Join Path

  1. Dealer starts a local room and displays a QR code.
  2. Players join the same local network or hotspot.
  3. Clients connect to the dealer host and receive a snapshot.
  4. Commands flow through the dealer until the room ends.

Reconnect Path

  1. Client receives a current projected snapshot.
  2. Missed card moves are folded into that snapshot.
  3. Server replays missed social events: peek notices and relevant shows.
  4. Old animations are skipped if they are no longer useful.

Implementation Shape

Suggested Technical Stack

Client

Expo and React Native for iOS, Android, and web from one TypeScript codebase.

Animation

React Native Reanimated for dealing, flips, hand fans, pile movement, and gestures.

Cloud Transport

Socket.IO over WebSocket with room snapshots, command acknowledgements, and event replay.

Offline Transport

Dealer-hosted local WebSocket room over LAN or hotspot, joined by QR code.

Persistence

Postgres for cloud history and recovery. Dealer-hosted rooms use the dealer's local log as authority; peers may keep a recent tail only for diagnostics.

Privacy

Casual privacy through projected state now. Cryptographic privacy is explicitly out of v1.

Risks And Concerns

What Needs Careful Engineering

Dealer Device Dependency

Offline mode depends on the dealer phone staying awake, connected, and sufficiently charged. If the dealer phone dies in v1, the room ends and any buffered local events that never reached peers may be lost.

Dealer-Hosted Privacy Ceiling

In dealer-hosted mode, the dealer device runs the authority and can technically inspect all card values. The v1 guarantee is UI projection, not cryptographic secrecy from the host.

Privacy Semantics

Face-down cards must not leak through event payloads, logs, debugging panels, analytics, or reconnect snapshots. Peek history is game state, not telemetry for unrelated later queries.

Command Races

Alice taking a card and Bob giving the same card can arrive in either order. Expected-version checks should reject the stale command and force the losing client to reconcile.

Remote Take UX

Taking from another hand is acceptable in the physical-deck model, but remote play lacks table presence. Take-from-hand events need stronger notification treatment than ordinary table moves.

Connectivity Variance

Local networks, hotspots, captive portals, and platform permissions create uneven behavior. QR join and clear reconnect states will matter.

Animation Consistency

Clients should animate the same event sequence while computing positions locally. The app should be able to skip stale animations during reconnects.

Rule Ambiguity

Baraja is a deck simulator rather than one specific card game. The engine should stay generic while public events make socially questionable moves visible to the table.
