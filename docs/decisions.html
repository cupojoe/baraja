Tooling Decisions

Each tool was chosen to support a specific constraint of the Baraja problem. This page records what was selected, what was considered and rejected, and why each trade-off was made.

Foundation

One Repo, Coordinated Builds

The engine package is a shared dependency of both the server and the mobile client. The build toolchain must enforce that the engine is compiled before either consumer runs a typecheck or build — by convention is not enough.

Selected

npm workspaces + Turborepo

npm workspaces (built into Node/npm) hoists shared dependencies and links local packages via symlink. Turborepo adds a task graph: dependsOn: ["^build"] guarantees the engine compiles before the server or mobile app builds or typechecks. Turbo also caches build artifacts so unchanged packages never rebuild.

  • Zero extra infrastructure — ships with Node 22 + npm.
  • Task ordering is explicit and reproducible.
  • Remote cache can be added later without changing package structure.
Rejected

pnpm workspaces / Yarn workspaces

pnpm was not installed in the development environment. Yarn workspaces offer no material advantage over npm workspaces at this project scale. Neither was worth the installation friction on top of a Node 22 requirement already present.

  • pnpm: not installed; adds a setup step with no clear payoff.
  • Yarn: separate install, separate lockfile format, no gain here.
Rejected

Nx / Lerna

Nx adds project-level config files, executor plugins, affected-command infrastructure, and a caching daemon. Lerna's primary value was version publishing, which this project does not need. Both are considerably more configuration than three packages warrant.

  • Nx: significant config overhead for three packages.
  • Lerna: versioning and publishing tooling, not a build graph.

Why the build order matters

@baraja/engine exports TypeScript declaration files from dist/. If the server or mobile app imports it before dist/ exists, the typecheck fails with a module-not-found error. Turborepo's task graph makes this constraint impossible to violate accidentally.

Accepted trade-off

npm's hoisting strategy occasionally causes phantom dependency issues in large repos. At three packages this is not a real risk. If the repo grows significantly, migrating to pnpm strict hoisting is a one-day change.

Development flow

npm run build -w packages/engine once, then turbo dev in the root. Turbo watches engine source and rebuilds incrementally; server and mobile hot-reload from their own watchers.

Type System

Shared TypeScript Across All Three Packages

@baraja/engine defines the contract between the server and the mobile client: CardState, ZoneState, ViewerSet, every command, and every event. If either consumer disagrees with the engine's shape, it fails at compile time, not at runtime in the middle of a game.

Selected

TypeScript 6 (strict mode)

Strict mode throughout. composite: true enables TypeScript project references so the server and mobile app typecheck against engine source directly. The server uses NodeNext module resolution to match Node.js semantics; the engine and mobile use Bundler resolution for ESM compatibility with Metro and tsx.

  • Single language across all packages — no context switch.
  • Projection bugs are type errors, not runtime surprises.
  • Shared event and command shapes with zero serialization drift.
Rejected

JavaScript (no types)

Without static types, command and event shape mismatches surface only when the code runs. The visibility projection — deciding whether a card value should reach a given player — is exactly the kind of logic that fails silently in untyped code.

  • No cross-package contract: any rename is a silent breaking change.
  • Card value leaks through projection would not be caught at build time.
Rejected

Flow / runtime-only (Zod, Valibot)

Flow's ecosystem has contracted significantly and offers no advantage over TypeScript here. Runtime validation libraries (Zod, Valibot) are useful at system boundaries but are not substitutes for static typing across the whole codebase.

  • Flow: smaller ecosystem, no material benefit over TypeScript.
  • Runtime-only validation: catches errors later and only on the happy path.

The key type: ViewerSet

type ViewerSet = "everyone" | Set<PlayerId> — the projection layer checks this before including a card value in any outgoing payload. TypeScript enforces that every code path handles both cases explicitly. Without this, a default-truthy check could silently broadcast private card values.

TypeScript 6 vs 5.x

TypeScript 6 is the current stable release. The project uses no deprecated patterns from 5.x, so there is no migration cost. The Expo toolchain (Metro, babel-preset-expo) resolves TypeScript via the Babel transform and is version-agnostic for type-stripping purposes.

Accepted trade-off

TypeScript 6 is very new. If a dependency exposes an incompatibility with the new compiler, pinning to 5.8.x in that package's devDependencies resolves it without touching the shared base config. The root tsconfig.base.json settings are all stable across both versions.

Client Platform

One Codebase for iOS, Android, and Web

Baraja needs to run on phones. The UI must handle touch gestures, card animations at 60 fps, and a persistent WebSocket connection — on both iOS and Android from the same source.

Selected

Expo SDK 55 + React Native 0.85 + Expo Router v4

Expo SDK 55 manages the native build toolchain, OTA updates, and the dependency graph for native modules. React Native 0.85 uses the New Architecture (Fabric renderer, JSI) by default. Expo Router v4 adds file-based routing with typed routes, enabling deep links for room invites without manual navigation config.

  • Single TypeScript codebase for iOS, Android, and web.
  • New Architecture enabled by default — lower JS bridge overhead.
  • Managed workflow keeps native build infra out of the repo.
  • Typed routes catch broken deep links at compile time.
Rejected

Flutter

Flutter uses Dart. The game engine is TypeScript. Sharing types, commands, and events between a Dart client and a TypeScript server requires a code-generation step (protobuf, OpenAPI) that adds significant maintenance overhead for every schema change.

  • Language boundary: TypeScript engine cannot be imported directly.
  • Schema generation adds friction to every engine type change.
  • Dart is a meaningful context switch for a TypeScript-first project.
Rejected

React Native CLI / Capacitor

The bare React Native CLI requires manual native project maintenance (Podfile, Gradle files, native module linking). Capacitor wraps a web app in a WebView — the JavaScript bridge latency compounds with animation and gesture work, which is the heaviest part of this UI.

  • RN CLI: no managed build infra, more native churn per SDK upgrade.
  • Capacitor: WebView bridge adds latency to the animation-heavy card UI.
  • Neither has a routing layer comparable to Expo Router.

New Architecture matters here

React Native's New Architecture moves layout and gesture handling off the JS thread via JSI. For a card game where gestures (drag, fan, flick) trigger animations, this is the difference between a smooth experience and one that stutters when the event loop is busy processing a server event.

Expo Router and room invites

Room join links are a first-class product feature for both cloud rooms and local dealer-hosted rooms (via QR code). Expo Router's typed deep link support means baraja://room/[id] routes are statically validated — a renamed route segment is a build error.

Accepted trade-off

Expo's managed workflow limits direct native module access. Any feature requiring custom native code (e.g., mDNS for local room discovery) requires ejecting to a bare workflow or using a config plugin. This is an expected constraint for dealer-hosted mode and is deferred to a later milestone.

Card Motion

60 fps Animations on the UI Thread

Dealing, shuffling, flipping, and cutting cards must animate smoothly even while network events are arriving and the game state is updating. Animation work that runs on the JavaScript thread is vulnerable to pauses from socket message processing.

Selected

React Native Reanimated 4 + react-native-worklets

Reanimated 4 runs animation logic in worklets — small functions that execute on the UI thread via JSI, entirely bypassing the JS event loop. The react-native-worklets package (split from Reanimated in v4) provides the shared memory and thread-safe value model that both gesture handlers and layout animations depend on.

  • UI-thread execution: card motion never drops frames from socket events.
  • Shared values let the server event update state while animation plays.
  • Gesture recognition and layout animation share the same worklet model.
Rejected

React Native Animated API

The built-in Animated API can run on the native driver for simple transforms, but complex interpolations and gesture-linked animations fall back to JS-driven execution. For a card game with continuous gesture input and concurrent state updates, the JS-driven fallback is a real performance ceiling.

  • JS-driven path is a real risk for this animation complexity.
  • No worklet model: gesture and layout code cannot share thread-safe values.
  • Less expressive than Reanimated for multi-card choreography.
Rejected

Lottie / pre-baked animations

Lottie plays back pre-authored animation files. Card positions in Baraja are computed at runtime from the game state — which card is in which zone, how many cards are in a hand fan, where the target pile is on screen. These positions cannot be pre-authored.

  • Not suitable for dynamic, state-driven card positions.
  • No interaction with gesture recognizers or layout engine.
  • Good for decorative UI elements; wrong tool for game pieces.

The concurrent event problem

A deal animation starts when the authority emits a CardsDealt event. While the cards are flying, the next player may play a card, sending another event. Reanimated's shared value model lets incoming events update game state without interrupting or dropping the in-progress animation.

Worklets as a peer dependency

Reanimated 4 splits react-native-worklets into a separate package (~0.8.1 for RN 0.81–0.85). Both must be installed and the Babel plugin must be registered in babel.config.js for the worklet transform to apply. Missing either causes silent fallback to JS-thread execution.

Accepted trade-off

Worklets require the Babel plugin and add a compilation step. Build times increase slightly. The plugin must be the last item in the Babel plugin list. These are well-documented constraints and do not affect the runtime behavior once configured correctly.

Game Transport

Socket.IO for Cloud Room Communication

The cloud authority model requires ordered broadcast to a named set of players, command acknowledgement so the client knows whether its move was accepted, and automatic reconnection so a brief network drop does not end the game.

Selected

Socket.IO 4

Socket.IO's room primitive maps directly to a game room: socket.join(roomId) and io.to(roomId).emit() handle player membership and broadcast without custom routing. Built-in reconnection with exponential backoff preserves session state across drops. Acknowledgement callbacks let the server confirm or reject each command before the client updates its local state.

  • Room primitives match the game model directly — no custom membership layer.
  • Acknowledgements enable the expected-version command/reject pattern.
  • Automatic reconnection with configurable backoff.
  • Falls back to long-polling where WebSocket is blocked (captive portals).
Rejected

Raw WebSocket (ws / native)

Raw WebSocket provides a message pipe and nothing else. Rooms, reconnection, acknowledgements, fallback transports, and namespace isolation must all be built on top. That is Socket.IO, reimplemented without the benefit of years of production use.

  • No rooms: player membership tracking is bespoke infrastructure.
  • No acknowledgements: command confirmation requires a custom protocol.
  • No reconnection: session resume logic is entirely custom.
Rejected

tRPC WebSocket / managed push (Ably, Pusher)

tRPC's subscription model fits query/mutation patterns; game events are not subscriptions to a data feed — they are ordered, versioned state mutations that must be sequenced and rejected. Managed push services (Ably, Pusher) add vendor lock-in and move the projection logic to the server in a way that does not compose with the authority model.

  • tRPC: subscription semantics mismatch the command/event model.
  • Ably/Pusher: vendor lock-in, cost at scale, less control over projection.
  • Managed services make dealer-hosted mode impossible without a separate transport.

Acknowledgements and version rejection

When a client sends a command, it includes expectedVersion. The server's acknowledgement callback returns either the new version (accepted) or a rejection reason. The client holds off updating its local state until it receives this ack. Socket.IO acks are a first-class primitive — no extra protocol layer needed.

Reconnect and snapshot replay

On reconnect, the client re-joins its room and requests the current projected snapshot. Socket.IO's room membership persists on the server through a configurable grace period, so a brief drop does not lose the player's seat. Missed events are folded into the snapshot rather than replayed individually.

Dealer-hosted transport

Socket.IO is also the planned transport for dealer-hosted rooms — the dealer device runs a socket.io server locally, and other players connect to it over LAN. The same client-side socket.io-client works for both cloud and dealer-hosted rooms; only the server URL changes.

Cloud Server

Express 5 on Node.js 22

The cloud server is a thin authority layer. It receives commands, validates versions, mutates state, persists events, and broadcasts projections. It should not accumulate framework complexity.

Selected

Express 5 + Node.js 22

Express 5 adds native async error handling — thrown errors in route handlers propagate to the error middleware without wrapping in try/catch. Socket.IO attaches directly to an Express HTTP server with one line. Node.js 22 is the current LTS and is a hard requirement of React Native 0.85's build toolchain regardless of server choice.

  • Minimal surface area — the server is an authority, not an application framework.
  • Native async error propagation in Express 5 removes a common footgun.
  • Socket.IO + Express is the canonical integration pattern with abundant examples.
  • Node 22 LTS is already required by the RN 0.85 dev toolchain.
Rejected

Fastify

Fastify's JSON schema validation and serialization pipeline is valuable for REST-heavy APIs. For this server, most traffic is WebSocket — the REST surface is a health endpoint and room creation. Fastify's Socket.IO integration requires an adapter plugin with less documentation than the direct Express pattern.

  • WebSocket-heavy server derives little benefit from Fastify's serialization pipeline.
  • Socket.IO adapter adds config overhead with no payoff at this scale.
  • Fewer examples for the Fastify + Socket.IO combination.
Rejected

Bun runtime

Bun's native WebSocket support is compelling, but the React Native 0.85 build toolchain explicitly requires Node.js 22. Running the server on Bun while the mobile dev toolchain requires Node creates a split environment with no clear benefit for this use case.

  • RN 0.85 build tools require Node 22 — Bun does not eliminate Node from the repo.
  • Socket.IO on Bun has lower test coverage than the Node.js path.
  • Split runtime environments add cognitive overhead for a solo or small team.

Server surface area

The server exposes one REST endpoint (GET /health) and one Socket.IO namespace (/). Room creation, joining, and all game commands flow through Socket.IO events. Express exists only to provide the HTTP server that Socket.IO attaches to and to serve future REST endpoints for room management.

tsx for development

tsx watch src/index.ts runs TypeScript directly without a compile step in development, using esbuild under the hood. The server restarts on file changes. For production, tsc compiles to dist/ and node dist/index.js runs the output. No transpiler is needed at runtime in production.

Accepted trade-off

Express does not validate incoming Socket.IO event payloads. Command shapes are typed at the TypeScript level but not runtime-validated at the socket boundary. Adding a runtime validator (Zod) at socket event handlers is the planned next step before any public-facing deployment.

State Storage

SQLite Now, Postgres When Ready

Game events and room snapshots need durable storage. The development environment should require zero external processes. Production needs multi-reader support and crash recovery. The persistence layer must handle both without rewriting the schema.

Selected

Drizzle ORM + better-sqlite3 → postgres-js

Drizzle defines the schema in TypeScript once (src/db/schema.ts). Switching from SQLite to Postgres is a driver swap: replace better-sqlite3 with postgres and change the Drizzle client import. The schema file does not change. In development, the database is a single baraja.db file with WAL mode enabled — no process to start, no credentials to manage.

  • TypeScript-first schema: table shapes are typed throughout the server.
  • SQLite for dev: zero infra, one file, WAL for concurrent reads.
  • Postgres for production: driver-only change, no schema rewrite.
  • Drizzle Kit generates migrations from the TypeScript schema.
Rejected

Prisma

Prisma generates a Rust-backed query engine and a JavaScript client from a .prisma schema file. The codegen step adds latency to the dev loop, and the WASM bundle is a concern for edge or embedded deployments. Prisma's value is schema-driven client generation; Drizzle is lighter and SQL-closer for this use case.

  • Codegen adds a step to every schema change cycle.
  • Heavier runtime: Rust query engine binary per target platform.
  • Less flexible for raw SQL when needed for event log queries.
Rejected

Redis / raw SQL strings

Redis is optimized for key-value access and pub/sub. The game event log is an append-only ordered sequence that needs versioned reads and atomic version-gated writes — relational semantics, not key-value. Raw SQL strings lose the type safety that makes the projection layer safe; a typo in a column name is a runtime error.

  • Redis: not relational; ordering and version-gating are awkward to implement.
  • Raw SQL: no type contract between schema and query results.
  • Both require significantly more infrastructure or discipline than Drizzle.

Schema shape

Three tables: rooms (id, created_at, closed_at), game_events (id, room_id, version, type, payload as JSON, created_at), snapshots (id, room_id, version, state as JSON, created_at). The event log is append-only; snapshots are written periodically to bound replay cost on reconnect.

WAL mode in SQLite

PRAGMA journal_mode = WAL enables concurrent readers with a single writer — multiple Socket.IO handlers can read the current state while the authority writes a new event. Without WAL, SQLite uses exclusive write locks and concurrent reads block.

Migration path to Postgres

Replace better-sqlite3 with postgres in apps/server/package.json, update the Drizzle client import in src/db/client.ts, change drizzle.config.ts dialect from sqlite to postgresql, and set DATABASE_URL. No schema changes. The app is brittle on SQLite by design; switching is a deliberate future step.

Architecture Boundary

Physical Deck Core, Rule Modules Above

Baraja's natural trajectory is to support bespoke games on top of a shared deck. The line between the physical-deck engine and any future game rules is being drawn now, while the engine has no callers, because the cost of drawing it later compounds with every command and event added to the wire.

Selected

Closed engine, layered rule modules

The engine implements only physical operations on cards and zones. GameCommand and GameEvent are closed discriminated unions with no escape hatch. Rule modules are pure functions (event, projectedState) => GameCommand[] that run on the authority alongside the reducer; they observe applied events and emit further physical commands. Game-specific semantics ride on a metadata bag attached to zones and cards — added to the schema in v1 even though nothing consumes it yet.

  • Replay, projection, and per-viewer privacy stay properties of a single reducer.
  • Adding a new game is a layer addition, not a core change.
  • metadata on the schema avoids a painful retrofit through event payloads and the persisted log.
  • Authority-side execution keeps clients deterministic — no two observers disagree on what happened.
Rejected

Open command union with rule-emitted commands

Allow rule modules to extend GameCommand with game-specific variants (e.g. EndTurn, Score). The reducer then dispatches to a registry of rule-provided handlers.

  • Visibility projection cannot reason about commands it does not know.
  • Replay-completeness becomes a property of every rule module, not just the engine.
  • Wire schema and persisted event log become per-game shaped — incompatible across rule sets.
Rejected

Game rules merged into the engine

Skip the boundary and grow the engine to know about specific games directly: turn order, scoring, legality checks all baked into the core reducer. This is what most card-game codebases drift into.

  • Every new game widens the central command vocabulary and reducer hotspot.
  • The "deck simulator" identity dissolves — the product becomes one specific game.
  • Rules conflicts (variant A vs variant B) become reducer branches, not module choices.

What lands in v1

Only the boundary itself: closed command/event unions, a metadata: Record<string, unknown> field on ZoneState and CardState, and this document. No rule-module runtime, no module registry, no game modes. The engine ships as a deck simulator and nothing else.

The integration shape (for later)

A rule module is (event, projectedState) => GameCommand[], run on the authority after every applied event. It can read state and metadata but cannot mutate either; its only output is more physical commands fed back into the engine. This preserves the single-reducer property and keeps the event log canonical.

Accepted trade-off

A rule that genuinely cannot be expressed as a sequence of physical commands forces a primitive addition to the engine, not a workaround in the rule layer. This is intentional — it keeps the wire schema honest. If that pressure shows up frequently post-v1, it is a signal to revisit the boundary, not to add an escape hatch.
